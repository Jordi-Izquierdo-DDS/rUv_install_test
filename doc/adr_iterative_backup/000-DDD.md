## 2. The component-selection protocol

For any question *"what do I use for X?"*, the order is:

1. **Query `ruvector-catalog`** first (CLI or SKILL.md). It will either answer or point to the relevant crate/namespace.
2. **Query pi-brain** (`brain_search`) with the concept name. Weight hits by α. Look at α ≥ 2 memories first.
3. **Query gitnexus** on the upstream repos for the specific symbols / call graphs. This is the ground truth for what is implemented and what is stub.
4. **Read upstream source directly** at `/mnt/data/dev/_UPSTREAM_20260308/<repo>/`. Confirm the code is production-quality (not `todo!()`, not a trivial stub, not marked experimental, not simplified placeholder).
5. **Then** pick the component per this rule:

   > **Pick what actually works and is the best available — not stubs, not inefficient placeholder code, not overly-simplified wrappers. When two options exist, prefer the ORIGINAL mature implementation over newer wrappers still maturing.**

6. **Record the choice in a scoped ADR** (001, 002, …) with evidence: gitnexus query outputs, file:line citations, α values. If no option is mature, the ADR marks the concern as **deferred** with the revisit trigger.

### Operator experience-based default (use as prior, not as crutch)

Ruflo is the *new* coordination layer but **still immature**. When choosing between a ruflo-flavoured wrapper and the original upstream component, the original usually wins — unless gitnexus + source confirm the wrapper is production-quality.

**This is a prior, not a shortcut.** Every decision must still pass steps 1–5 above.

## 3. SCOPE — the full self-learning + self-improving cycle

Ruflo v4's domain scope is the **complete self-learning + self-improving cycle** of a Claude-Code agent. The scope has **three orthogonal axes** (often conflated; they are NOT the same thing):

- **Architectural layers** (coarse: FoxRef → Ruvector → Ruflo, but ruvector internally has many sub-layers — *"3 layers"* is only the outermost view)
- **Self-learning 3-TIER routing model** (reactive / adaptive / deliberative) — vertical routing of every inference request by complexity
- **3-LOOP temporal learning cadence** (A / B / C) — when learning work actually runs

Pi-brain α=13 memory ("SONA Three-Tier Learning Architecture") is the authoritative definition of self-learning; the 3-loop model (foxRef Part02 L116–196) is the authoritative definition of temporal cadence. They compose — they do not replace each other.

### 3.2 Self-learning: the 3-TIER inference routing model (pi-brain α=13)

Pi-brain α=13 ("SONA Three-Tier Learning Architecture") is authoritative:

> *"SONA implements a three-tier learning system composed of reactive, adaptive, and deliberative layers. Together these tiers enable cost-efficient inference routing where over 60% of requests never reach an LLM."*

Every inference request traverses these tiers **top-down until one can serve it**. Higher tiers are cheaper but narrower; lower tiers are broader but expensive.

| Tier | Mechanism | Latency / Cost | When it serves | Upstream anchor |
|---|---|---|---|---|
| **Reactive** | Cached pattern matching via WASM-compiled rules; **bypasses LLM entirely** | sub-millisecond; $0 | cache hit on a known pattern; routine transforms | `SonaEngine::findPatterns` + precompiled MoE routing gates |
| **Adaptive** | Online gradient updates with **MicroLoRA deltas** (rank 1–2) on top of recent interactions | low; cheap model call | adaptable novelty — similar to something seen, but needs a fresh delta | `sona::MicroLoRA` (`crates/sona/src/lora.rs`) + `AdaptiveEmbedder.learnFromOutcome` |
| **Deliberative** | Full LLM reasoning (Sonnet / Opus) | high; full inference cost | task complexity > 30%; genuinely novel cases | Claude via Claude-Code; `LearnedRouter` (`ruvector-attention::LearnedRouter:29`) decides escalation |

**Economic property:** *"over 60% of requests never reach an LLM"*. Ruflo v4's responsibility is to **not block** this routing — every hook must let the reactive tier try first. If ruflo adds pre-LLM work beyond thin adapter logic, it undermines this economic guarantee.

**Hook-to-tier mapping (what tier each Claude-Code event is served by):**

| Claude-Code hook | Typical tier | Rationale |
|---|---|---|
| SessionStart / SessionEnd | reactive | pure bookkeeping |
| PreToolUse / PostToolUse | reactive (cache) → adaptive (if miss) | step reward + step embedding; reactive tier handles the recording |
| UserPromptSubmit | reactive (pattern lookup) → adaptive → deliberative | context retrieval is reactive; adaptive applies LoRA delta; if complexity > 30%, deliberative Claude handles the turn |
| Stop / SubagentStop | adaptive (Loop-B learning triggered) | runs after the deliberative work; triggers learning cadence |

### 3.3 Self-learning: the 3-LOOP temporal cadence (foxRef Part02 L116–196)

Self-learning = **accumulating knowledge** from interaction (trajectories → patterns → memory). Three concurrent state machines running on the same trajectory buffer at different cadences.

| Loop | Cadence | Purpose (self-learning work) | Canonical upstream symbol |
|---|---|---|---|
| **A — Instant** | per inference (<10ms) | capture trajectory step; apply MicroLoRA delta (rank 1–2); attach query embedding | `sona::LoopCoordinator::on_inference` (`crates/sona/src/loops/coordinator.rs:12`) |
| **B — Background** | every ~30s (<100ms p99) | k-means over trajectory buffer → extract patterns → BaseLoRA per-layer adapt → mincut/GraphMAE refinement → store in HNSW | `LoopCoordinator::maybe_run_background` + `ruvllm::EpisodicMemory::extract_patterns:309` + `ruvector-gnn::GraphMAE:271` + `ruvector-mincut-gated-transformer::SparseMask:97` |
| **C — Session-End** | on SessionEnd (no budget) | EWC++ Fisher consolidation; MemoryCompressor (10:1); pattern prune; metrics export | `sona::EwcPlusPlus::consolidate` (`crates/sona/src/ewc.rs:65`) + `ruvllm::MemoryCompressor` (`crates/ruvllm/src/context/episodic_memory.rs:160`) |

**Hook mapping (L3 → L2):**

| Claude-Code hook | Loop served | Canonical call |
|---|---|---|
| SessionStart | boot | `HooksIntegration::session_start` + singletons init |
| UserPromptSubmit | A (capture prompt + retrieve) | `pre_task` → context block returned |
| PreToolUse | A (pre-step) | `pre_edit` + pre-bash regex (ruflo-only) |
| PostToolUse | A (post-step) + possibly trigger B | `post_edit` + `maybe_run_background` |
| Stop / SubagentStop | B (judge + distill + refine + store + tune) | `post_task` fires whole Loop B |
| SessionEnd | C (consolidate + forget + prune + export) | `session_end` with `exportMetrics` |

**Full stage-by-stage mapping:** `doc/reference/visual-summary_Phase3_proposal.html` and § 1 of `doc/reference/foxref-architecture-guide.md`.

**How tiers and loops compose:** tiers describe **WHERE** a request is served (routing axis); loops describe **WHEN** learning happens on the trajectory buffer (temporal axis). A single hook event may ride the reactive tier for its own serving AND feed a trajectory step that Loop A records AND triggers Loop B if the background gate is met. The two axes are orthogonal and both must be honoured.

### 3.4 Learning-cycle phases × axes (the unified table)

The 8 phases of the learning cycle are the **most important** view — they describe what actually happens on each turn. Each phase belongs to one loop (temporal), may ride one or more tiers (routing), is triggered by a specific Claude-Code hook, and calls a canonical upstream symbol.

| # | Phase | Loop | Tier(s) served | Hook trigger | Canonical upstream symbol | file:line |
|---|---|---|---|---|---|---|
| 1 | **CAPTURE** (record trajectory step) | A | reactive | `PreToolUse`, `PostToolUse` | `SonaEngine::addTrajectoryStep` | `crates/sona/src/loops/coordinator.rs:12` (via `LoopCoordinator::on_inference`) |
| 2 | **RETRIEVE** (find similar patterns) | A (same tick as capture) | reactive → adaptive (if miss) | `UserPromptSubmit` | `SonaEngine::findPatterns(emb, k)` + `ReasoningBank::findSimilar` | `@ruvector/sona` + `crates/ruvllm/src/reasoning_bank/mod.rs` |
| 3 | **APPLY** (route + LoRA delta) | A | adaptive or deliberative (by complexity) | `UserPromptSubmit` | `LearnedRouter::route` + `MicroLoRA::forward_layer` | `crates/ruvector-attention/src/moe/router.rs:29` + `crates/sona/src/lora.rs` |
| 4 | **JUDGE** (verdict on outcome) | B | — (not a serving tier) | `Stop` / `SubagentStop` | `VerdictAnalyzer::analyze(&trajectory)` | `crates/ruvllm/src/reasoning_bank/verdicts.rs:314` |
| 5 | **DISTILL** (k-means over trajectories + refinement) | B | — | `Stop` (Loop-B gate tripped) | `EpisodicMemory::extract_patterns` + `GraphMAE` + `SparseMask` | `crates/ruvllm/src/context/episodic_memory.rs:309` + `crates/ruvector-gnn/src/graphmae.rs:271` + `crates/ruvector-mincut-gated-transformer/src/sparse_attention.rs:97` |
| 6 | **STORE** (insert pattern in HNSW) | B | — | `Stop` (after DISTILL) | `PatternStore::insert(pattern)` | `crates/ruvllm/src/reasoning_bank/pattern_store.rs` |
| 7 | **REINFORCE** (per-dim EMA + witness audit + regime update) | B | — | `Stop` (after STORE) | `QualityScoringEngine::track` + `WitnessSource::record` + `RegimeTracker::update` | `crates/ruvllm` + `crates/prime-radiant/src/ruvllm_integration/witness.rs:92` + `crates/prime-radiant/src/sona_tuning/tuner.rs:36` |
| 8 | **FORGET** (EWC Fisher consolidation + prune + compress) | C | — | `SessionEnd` | `EwcPlusPlus::consolidate` + `ReasoningBank::prune` + `MemoryCompressor::compress` | `crates/sona/src/ewc.rs:65` + `crates/sona/src/reasoning_bank.rs:47` + `crates/ruvllm/src/context/episodic_memory.rs:160` |

**Reading the table:**
- **Phases 1–3** (CAPTURE → RETRIEVE → APPLY) happen per-inference in Loop A. These are the tier-routed phases — most requests stay in reactive; adaptive handles novelty; deliberative handles complexity > 30%.
- **Phases 4–7** (JUDGE → DISTILL → STORE → REINFORCE) fire in Loop B after a trajectory closes on `Stop`. Not tier-routed because they're learning work, not serving work.
- **Phase 8** (FORGET) is Loop C on `SessionEnd`. Last-chance consolidation before daemon may die.

**Traceability rule:** every L3 (ruflo) hook handler must trace to at least one phase in this table. If a hook fires without mapping to a phase, it's either dead code or an undocumented extension — both need explanation. Conversely, every phase in the table must have an upstream symbol verified via the §2 protocol before being called canonical.

**Full per-stage details + IPC command mapping:** `doc/reference/visual-summary_Phase3_proposal.html` (14-stage view — expands phases 1/5 with sub-stages BOOT, APPLY-route/LoRA split, REFINE, STORE+TUNE, CONSOLIDATE, PRUNE, EXPORT).

### 3.5 Self-improving: the system gets better over time

Self-improving = **the engine itself improves**, independent of specific knowledge (weights adapt, routing gets smarter, embedder reflects outcomes, drift gets detected, the system tunes its own hyperparameters).

| Concern | Mechanism | Canonical symbol |
|---|---|---|
| LoRA adaptation (weights improve per trajectory) | MicroLoRA rank 1–2 per inference; BaseLoRA per Loop-B tick | `sona::{MicroLoRA, BaseLoRA}` (`crates/sona/src/lora.rs`) |
| Embedder adaptation (embedding space tunes itself) | `AdaptiveEmbedder.learnFromOutcome` after each outcome feedback | `ruvector` npm (`require('ruvector').AdaptiveEmbedder`) |
| Router learning (MoE expert selection improves) | SONA-trainable weights on the router | `ruvector-attention::LearnedRouter` (`crates/ruvector-attention/src/moe/router.rs:29`) |
| Catastrophic-forgetting prevention | EWC++ Fisher matrix protects important weights during new learning | `sona::EwcPlusPlus` (`crates/sona/src/ewc.rs:65`) |
| Pattern quality tracking (per-dim EMA + trend) | `QualityScoringEngine` tracks schema/coherence/diversity/temporal/uniqueness | `ruvllm::QualityScoringEngine` |
| Regime tuning (system-wide thresholds adapt) | `RegimeTracker` adjusts SONA hyperparams in response to regime shifts | `prime-radiant::RegimeTracker` (`crates/prime-radiant/src/sona_tuning/tuner.rs:36`) |
| Witness audit (every decision is traceable) | SHAKE-256 witness chain on verdicts | `prime-radiant::WitnessSource` (`crates/prime-radiant/src/ruvllm_integration/witness.rs:92`) |
| Drift detection (system notices it's degrading) | Delta Drift / CUSUM drift monitors | `ruvector-coherence`, `prime-radiant` |
| Collective learning (pattern sharing via pi.ruv.io) | Share verified α ≥ 2 patterns with the π network; pull validated ones back | `@ruvector/pi-brain` MCP client (orthogonal external layer) |

**Self-learning and self-improving are two halves of the same cycle:** Loop A captures experience (§3.4 phases 1–3); Loop B extracts patterns and updates weights (§3.4 phases 4–7); Loop C consolidates and protects (§3.4 phase 8). The 3 tiers (reactive/adaptive/deliberative) then SERVE from that accumulated state. The system gets smarter (self-learning) AND better at being smart (self-improving) from the same event stream, and both benefits flow through the reactive-first tier routing so costs stay bounded.

### 3.6 What ruflo v4 (L3) is responsible for — and explicitly NOT

**L3 IS responsible for** (adapter layer only):

- Parsing Claude-Code hook events (stdin/JSON format)
- Stripping `<task-notification>` / `[INTELLIGENCE]` self-talk
- Pre-bash regex shell-safety (no upstream analog; stays)
- Warm daemon lifecycle + IPC transport (UDS, JSON-line)
- Forwarding each hook event to the correct upstream call (table in § 3.2)
- Reading `ranked-context.json` (Claude-Code auto-memory PageRank output)
- Formatting the `[INTELLIGENCE]` block for Claude-Code injection

**L3 is NOT responsible for** (delegated to L2 upstream):

- Trajectory shape construction → `TrajectoryRecorder` / `QueryTrajectory`
- Pattern building → `Pattern::from_trajectory`
- Confidence formulas → upstream decides from reward
- Step-type classification → `stepTypeToPatternType` (upstream)
- k-means / HNSW / EWC / LoRA → never reach for these from JS
- Memory persistence — backend selection happens in L2 (see OQ-1)
- Retrieval scoring / interleave policies beyond trivial merge → upstream returns ranked context

**If any logic from the "NOT" list creeps into L3, it's a violation of §3.1's direction-of-causality rule and must be reverted.**

## 4. Domain & bounded contexts

| # | Context | Concern (within §3 scope) | Canonical (today) | Status |
|---|---|---|---|---|
| **C1** | Claude-Code adapter | stdin/stdout hook format, event dispatch, pre-bash safety, IPC transport | `.claude/helpers/hook-handler.cjs` | stable — Claude-Code-specific, no upstream analog |

Contexts C1, C5 are **local adapter layers** (Claude-Code format). C2, C3, C4 are **upstream delegation** (we wrap, never reimplement). C6, C7 are **external services**.

**Boundary rule:** never shape domain logic inside C1/C5. If a helper in `.claude/helpers/` starts growing pattern-building, confidence formulas, or step-type heuristics, it's leaking out of its adapter context and must be pushed back into upstream (C2/C3/C4).

## 5. Standing decisions (today)

### D1 — No invention
The only valid code is what upstream returns: direct calls, no extra logic. If upstream doesn't produce the expected result with canonical calls, **report upstream**, do not patch around it. (Operator 2026-04-13.)

### D2 — No RVF (see ADR-001)
RVF cognitive containers deferred — too experimental upstream. Witness-chain needs route through `prime-radiant::WitnessSource`, not RVF.

### D3 — No local Rust build (see ADR-002 RESOLVED)
No Cargo path-deps on local workspace trees. All NAPI/WASM comes via published npm.

### D4 — Ruflo immature → prefer originals
Default bias: when two mature components exist and one is a ruflo-flavoured wrapper, prefer the original. Override only with gitnexus+source evidence the wrapper is production.

### D5 — AGENTDB-VS-RUVECTOR.md is reference-only
Not a spec. Useful as initialization pattern hint; must not cite as authority. All memory/persistence decisions must independently verify via gitnexus + source.

### D6 — L3 ruflo cap: ~850 LOC total
`verify.sh` enforces this. The cap exists because L3 should only be adapter-layer. Exceeding the cap signals logic leaking from L2 into L3.

## 7. Open questions (to be resolved via protocol, then recorded in scoped ADRs)

### OQ-1 — C4 Memory layer — which is production-mature and which is stub?

**State (2026-04-14):** Candidates identified — `agentdb@3.0.0-alpha.11` (npm standalone) OR AgentDB bundled in `@claude-flow/memory` OR `@ruvector/rvlite` OR something in `ruvector-postgres`. Also: `ReasoningBank` appears as a class inside `agentdb` AND in `@ruvector/ruvllm` AND in the `sona` crate — three separate implementations of the same name.

**To verify before deciding (per §2 steps 3–4):**
1. gitnexus: which `ReasoningBank` is production and which is stub?
2. source: is `agentdb@3.0.0-alpha.11` code mature or early-alpha scaffolding?
3. source: does `@ruvector/rvlite` handle structured records (not just vectors)?
4. source: does `ruvector-postgres` publish an npm client or is it crate-only?
5. source: is there a production ReasoningBank implementation inside the `sona` crate (not `ruvllm`) that is NPM-reachable?

**Tradeoff to weigh (per D4):** agentdb description claims "150x faster than SQLite with self-learning GNN, 6 cognitive memory patterns" but it's alpha.11; originals may be more mature. Prefer originals — but only after verification confirms they are actually more mature.

**Exit:** record decision in ADR-003 (memory layer) with `gitnexus ✓ source ✓` evidence.

### OQ-2 — Upstream SonaEngine "insufficient trajectories"

`SonaEngine::forceLearn()` returns `"skipped: insufficient trajectories"` with canonical usage. Default `pattern_clusters=100` per `crates/sona/src/types.rs:387`. Is this a hard requirement, a soft hint, or a bug?

**To verify:** read `engine.rs` / `loops/coordinator.rs` / `loops/background.rs` for the actual gate in `extract_patterns`. If it's `trajectories < pattern_clusters` → document as upstream behaviour, ruflo accepts the accumulation phase. If it's a different condition → record the finding and adapt.

**Rule:** do NOT override `pattern_clusters` via `SonaEngine.withConfig` unless source-verified the override is the canonical fix.

### OQ-3 — SonaEngine persistence (depends on OQ-1)

Published `@ruvector/sona@0.1.5` exposes no `saveState`/`loadState`. Options: (a) event-sourcing via C4 memory layer once OQ-1 resolved, (b) accept in-memory only, (c) report upstream as missing feature.

**Exit:** depends on OQ-1. If C4 chosen component supports trajectory replay, go (a). Otherwise (b) + explicit README note.

## 8. Process invariants

1. Every new dependency added to `package.json` must cite the ADR that approved it.
2. Every memory/feedback file with a prescriptive "use X" rule must cite the source (gitnexus query + file:line, or α ≥ 2 memory) that justified it.
3. `scripts/verify.sh` enforces the adapter-layer LOC caps and the "no-v3-era-reinvention symbols" gate. New gates can be added; existing ones cannot be silently relaxed.
4. When the operator corrects a decision (e.g. "PROHIBIDO INVENTAR"), the correction becomes a **feedback memory** with explicit quote + date; the affected code reverts or re-aligns; the ADR that granted the prior decision is amended or marked superseded.
5. Before claiming a component works, it must pass the §2 protocol including source verification that it is **not a stub**.
